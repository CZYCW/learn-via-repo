# Teaching Artifact Directory Structure Spec

> All teaching artifacts generated by learn-via-repo must follow the directory skeleton defined here. The structure is grounded in Diátaxis (the four-quadrant documentation framework) + Cognitive Apprenticeship (6 stages) + Bloom taxonomy (cognitive levels) + IA wayfinding principles. See `theory.md` for the theoretical basis.

> **Note**: artifact contents (welcome messages, READMEs etc) are templated in English in this doc, but the actual generated artifacts adapt to the user's language. If the user spoke Chinese during Phase 0/1, the artifacts come out in Chinese. See SKILL.md's "Language of Generated Teaching Artifacts" rule.

## §2. Responsibility of each directory

### §2.1 README.md (root)

**Responsibility**: the entry point when a student first opens the teaching artifacts.
**Diátaxis type**: Wayfinding (not in the four quadrants but a core IA principle).
**Tone**: **encouraging, lowers fear, plain language** (V3 strengthened — this skill is for beginners; first impression decides whether the student continues).

#### Template structure (V3 strengthened)

```markdown
# Welcome to your {domain} learning journey 🚀

You're going to systematically learn {domain} by deeply reading [**{target repo}**]({repo URL}).

> **First thing**: you don't need to understand {domain} yet — that's exactly what this teaching is here to fix.
> I (your AI teacher) will take you from zero to independently tracing the code, using a structured methodology.

---

## How this teaching works

**5 stages + 4 cross-cutting resources** (you don't need to read everything — just follow the recommended path):

{Directory visualization (per §1 skeleton)}

**Recommended learning path**:

{Two templates based on whether Phase 2.5 generated stage-0: A includes stage-0 / B skips stage-0}

1. **stage-0 fundamentals** (only if generated): get hands-on first — run a simple repo, modify it, play with it
2. **stage-1 foundations**: no code reading; build the domain mental model for {domain}
3. **stage-2 architecture**: read {target repo}'s design principles (architecture diagrams + ADRs); no deep code yet
4. **stage-3 walkthroughs**: I walk you through {target repo}'s core code
5. **stage-4 applied** (only if you provided a project PRD): apply what you've learned to your own project

**Cross-cutting resources** (use anytime, no order required):
- 📚 `reference/` — facts (code-map / data-flow / glossary)
- ✅ `exercises/` — verify your understanding with problems (paired with each walkthrough)
- 📖 `resources/` — external authoritative materials (Tier 1/2/3)
- ⚙️ `meta/` — methodology + progress tracking

---

## How I'll teach you

- **Never skip the basics**: A/B-level beginners go through stage-0 first — reading code without context is brutally abstract
- **Use Feynman checks**: "I get it" isn't a pass — you have to explain it in one sentence with no jargon, proving you've actually internalized it
- **Every code reference has file:line**: you can open exactly the line I'm pointing at, not an abstract description
- **Always start from one request's path**: not from `entry.ts` / `index.ts` (which is just init code)
- **No improvising code**: every "minimal implementation" references real, runnable code — never AI-improvised code that might not compile

For the full methodology, see `meta/teaching-style.md` (5-minute read).

---

## What I need from you

| Phase | Your action |
|---------|---------|
| Concept explanation | After I'm done, rephrase it **in your own words** (Feynman check). If you can't, that means you don't really get it — tell me |
| Before going to code | I'll ask you to "open {file}, find {function}" — you actually do it; if you can't find it, say so |
| Code walkthrough | Follow along — **don't fake understanding**; if you're stuck, interrupt me right away |
| Industrial enhancements | Listen to me explain "the extra N lines do this" — remember: "not more concepts, more robust" |
| End of lesson | Do the matching `exercises/` problems; if you got something wrong, go back and review |

---

## What to do when you're stuck

**Don't push through alone** — being stuck is part of learning. Specific strategies:

| Situation | What to do |
|------|-------|
| "I didn't get the concept" | Just say "I didn't get that, try another angle" — I'll switch to a concrete example or reverse argument |
| "I can't find the code" | Say "I can't find {function}" — I'll give an exact line + teach you grep / IDE jump |
| "stage-0 hands-on is stuck with an error" | If you're stuck ≥ 15 minutes, speak up — I'll run 5W diagnostics (what did you run? expected? actual? ...) |
| "I got the exercise wrong" | Say "I got it wrong, which section to review?" — I'll point you to the right walkthrough block |
| "Pace is too fast" | Say "slow down" — I'll break it into smaller chunks |

---

## When to start

Say `start stage-0` (if available) or `start stage-1` to begin.
You can also browse the artifacts freely first. Start the lesson when you're ready.
```

#### Required elements checklist

- ✅ Encouraging opening (the first line lowers fear)
- ✅ One sentence saying "this teaching is about which repo, which domain, aimed at what goal"
- ✅ Visualization of the full directory skeleton
- ✅ Recommended learning path (two templates based on whether stage-0 was generated)
- ✅ Note on whether stage-0 was generated
- ✅ When to use the cross-cutting resources
- ✅ **"How I'll teach you"** (student-perspective overview of the methodology)
- ✅ **"What I need from you"** (student-side action guide)
- ✅ **"What to do when stuck"** (user-perspective version of Scenario D unblocking)
- ✅ "When to start" trigger phrase

### §2.2 meta/

**Responsibility**: teaching meta-info — methodology, path, progress.
**Generation timing**: first in Phase 3 (other directories depend on meta/learning-path.md's path definitions).

- **teaching-style.md**: a student-readable version derived from `methodology.md`. When a student questions why a Feynman check is so strict, they come here for the rationale.
- **learning-path.md**: recommended order + graduation criteria per stage. Verification requirements at each stage transition (e.g., "before stage-3, you must be able to verbally summarize stage-2's 4 core architecture diagrams").
- **progress.md**: student progress tracking file; template includes "current stage / done / stuck on / next start point." Updated at the end of every lesson.

### §2.2b stage-0-fundamentals/ (added in V2)

**Responsibility**: hands-on prerequisite — before reading stage-1/2/3, the student clones an authoritative beginner repo, runs it, modifies it, plays with it, building "I've actually written one of these" muscle memory.
**Diátaxis type**: Tutorial (learn by doing).
**Bloom level**: Apply (hands-on first = applying).
**Constructionism support**: Papert's "learning by building" — doing is 10x more effective than passive listening. See `theory.md` §13.
**Generation timing**: only when Phase 2.5 assesses the student as A/B/C/D level; E level skips.

#### File list

- **README.md**: hands-on checklist — clone commands, recommended chapter order, expected time, what to do when stuck
- **prerequisites.md**: entry conditions
  - Language runtime versions (depend on the recommended repo, e.g., Python ≥ 3.10 / Node ≥ 18 etc)
  - **LLM inference setup**: based on what the student picked in Phase 1.6 (remote hosted / local), write the matching config steps
  - Environment variables / config file examples
  - **Principle**: LLM inference style is a config preference; it doesn't affect learning (see `forbidden-list.md` §5b meta-lesson 1)
- **recommended-repos.md**: 1-3 beginner repos **vetted by Phase 2C + confirmed by the user**
  - For each repo: URL, star count, why it's authoritative, relevance to the target repo
  - **AI doesn't unilaterally pick** (see `forbidden-list.md` §5 meta-lesson 3)
- **learning-guide.md**: for each recommended repo
  - Which chapters are stage-3 §4.1 reference sources (must read)
  - Which chapters are supplementary (skippable)
  - Recommended reading order (if there are multiple repos)
- **graduation-check.md**: what you should be able to do after (verifiable)
  - "Can you write the act() recursion independently without looking at code?"
  - "How many demos did you run?"
  - "Can you change the system prompt and observe the agent's behavior change?"
  - Passing = the prerequisite for entering stage-1

#### Relationship to other directories

- **stage-0 → stage-3**: code from stage-0 is the **preferred** reference source for stage-3 §4.1 ("the code the student knows best")
- **stage-0 → reference**: stage-0's specific file:line doesn't go into code-map.md (code-map.md indexes the **target repo**, not the beginner repo)
- **stage-0 → exercises**: stage-0's verification questions go into graduation-check.md, **not** into the three exercises/ files (exercises/ is paired with stage-3)

#### Key constraints (see forbidden-list §5)

- **First-principles constraint**: recommended repos must have no agent-loop abstraction wrapping — skeleton like `messages.append(...)` + `if stop_reason == "tool_use"` must be visible. See `forbidden-list.md` §5 for the judgment method + disqualifying-framework list
- **LLM inference style doesn't affect selection**: both remote-hosted and local repos are qualified — the agent loop skeleton is decoupled from the LLM call
- **Teaching value ≠ authority**: reference-type resources (cookbook / quickstart / production demo) are not suitable as stage-0 main line, even if authoritative — go into reference/. Judgment: does the README have a progression order ("first N, then N+1")?

#### Real recommendation for the OpenClaw learning scenario (example, vetted via Phase 2C + user evaluation)

```markdown
# stage-0-fundamentals/recommended-repos.md (OpenClaw learning scenario example)

1. **rohitg00/ai-engineering-from-scratch** Phase 14, first 12 lessons (7.8k stars)
   - Systematic course structure, "raw API first then framework"
   - Skip lessons 13-18 (they introduce LangGraph)

2. **pguso/ai-agents-from-scratch** (JS, 3.5k stars)
   - 14 lessons, most systematic progression; JS aligns with OpenClaw TS
   - Default local LLM; can swap to OpenAI API

3. **Leonie Monigatti notebook + blog post**
   - 4-component progression (LLM / memory / tool / agent loop), Anthropic API
   - Same Anthropic API family as OpenClaw — seamless transition
```

### §2.3 stage-1-foundations/

**Responsibility**: domain-wide cognition — **no code reading**, build the mental model.
**Diátaxis type**: Explanation.
**Bloom level**: Remember + Understand.

- **01-mental-model.md**: the "five-element abstraction" of this domain (e.g., AI agent = LLM + Memory + Tools + Loop)
- **02-design-patterns.md**: 3-5 core design patterns in this domain
- **03-state-of-the-art.md**: industry status + frontier (based on web research)

**Key constraint**: no repo file:line appears. After this stage, the student should have "industry-practitioner-level" cognition of the domain — they can discuss it independently.

### §2.4 stage-2-architecture/

**Responsibility**: target repo's architecture — **no deep code details**, build the architecture-layer mental model.
**Diátaxis type**: Explanation.
**Bloom level**: Understand + Apply.

- **00-overview.md**: architectural bird's-eye view (mermaid + four-pillar summary)
- **01-{module}.md** ~ **N-{module}.md**: one per module

Recommended per-piece structure (not strict — distinct from walkthrough-template):
- §0 General domain view (external material + how the domain handles this problem)
- §1 What problem does this module solve
- §2 Core concepts + glossary
- §3 Overall architecture (mermaid included)
- §4 Key design decisions (ADR format: Context / Decision / Consequence)
- §5 Runtime scenario (mermaid sequence diagram, 1 typical use case)
- §6 Code map (high-level, 3-5 core file:line, **don't expand**)
- §7 Implications for the user's target project

**Key constraint**: file:line references are allowed, but only as "pointing to where it is," not "line-by-line explanation." Line-by-line goes into stage-3-walkthroughs.

### §2.5 stage-3-walkthroughs/

**Responsibility**: instructor walks through code — strictly the Minimal Comparison Method, five steps.
**Diátaxis type**: Tutorial.
**Bloom level**: Apply + Analyze.
**Cognitive Apprenticeship**: Modeling (teacher demos) + Scaffolding (the Minimal Comparison Method is the scaffolding).

Every piece must strictly follow the §0-§8 nine-section template in `walkthrough-template.md`.

**Generation order**: go module-by-module along the data flow. Piece 1 must be "one message end-to-end" (global flow sense); piece 2 onward splits by module.

### §2.6 stage-4-applied/

**Responsibility**: apply the repo patterns to the user's target project.
**Diátaxis type**: How-to + Explanation mix.
**Bloom level**: Create.
**Cognitive Apprenticeship**: Exploration (student self-directed).

- **prd.md**: user project's PRD (copy directly if Phase 1 provided one; otherwise leave a template)
- **reusable-patterns.md**: which of the repo's designs you can directly reuse (with three columns: repo's design / how to use it in user's project / why it fits)
- **what-needs-change.md**: which of the repo's designs don't fit the user's scenario, how to change them (three columns: repo's design / why it doesn't fit / how to change)
- **mvp-sketch.md**: MVP design sketch for the user's project — assembled from patterns learned in stage-1/2/3

**Key constraint**: content in this directory **is only generated when the user provided a target project PRD**. Otherwise just put a README.md explaining "how to apply what you've learned."

### §2.7 reference/ (cross-cutting)

**Responsibility**: fact lookup; doesn't explain, doesn't teach.
**Diátaxis type**: Reference.
**When to use**: when the student forgets a file:line during a stage-3 walkthrough, they come here.

- **code-map.md**: index of core file:line across the whole repo, organized by module. Each entry has a one-sentence description.
- **data-flow.md**: end-to-end path diagram of one message from entry to LLM call (mermaid + file:line).
- **glossary.md**: glossary — domain general terms + repo-specific terms, one-sentence definition each.

### §2.8 exercises/ (cross-cutting)

**Responsibility**: check whether the student really understood.
**Generation timing**: every time you finish a stage-3-walkthroughs/{N}-{topic}.md, immediately append the topic's questions to all three files (see `exercises-template.md`).

- **concept-checks.md**: Feynman-style ("explain in one sentence, no jargon" + reverse argument). 3 per topic.
- **code-finding.md**: file:line location questions (verifiable). 3 per topic.
- **transfer.md**: transfer to the user's project (open-ended but with direction). 2 per topic.

Questions are organized under `## {N}-{topic}` headings, one-to-one with walkthroughs.

### §2.9 resources/ (cross-cutting)

**Responsibility**: index and archive of external materials.
**Diátaxis type**: Reference.

- **README.md**: Tier 1/2/3 tiered index
  - Tier 1 (must read): domain-authoritative official guides, widely cited papers, domain-authoritative deep articles
  - Tier 2 (deep): high-quality technical blogs, specific implementation tutorials, framework official docs
  - Tier 3 (reference): supplementary materials, case studies, competitor analysis
- **papers/**: paper PDFs (when downloadable)
- **articles/**: high-quality blogs (download to markdown via WebFetch)
- **videos.md**: video link list (YouTube / Bilibili etc)

## §3. Cross-references between artifacts

```
README.md (root)
  ├─→ meta/learning-path.md (path)
  ├─→ stage-1 / stage-2 / stage-3 / stage-4 (in order)
  └─→ reference / exercises / resources (cross-cutting, anytime)

stage-1-foundations/*.md
  └─→ resources/ (cites Tier 1/2 material)

stage-2-architecture/*.md
  ├─→ stage-1 (mental-model prerequisite)
  ├─→ reference/code-map.md (cite only, don't expand)
  └─→ resources/

stage-3-walkthroughs/{N}-{topic}.md
  ├─→ stage-2 same-topic doc (architecture layer)
  ├─→ §6 test questions link → exercises/{three types}.md#{N}-{topic}
  ├─→ §4 code → reference/code-map.md (cite precise location)
  ├─→ §0 general view → resources/ (cite Tier 1 material)
  └─→ §7 implications for your project → stage-4-applied/

stage-4-applied/
  └─→ cites all same-topic stage-3 patterns
```

## §4. File naming conventions

- Stage directories use `stage-{N}-{name}/`, N is 1-4
- Files within each stage use `{two-digit number}-{english-short-title}.md`
- Cross-cutting directories don't use numbers (e.g., `reference/code-map.md`, not `01-code-map.md`)
- README.md always lowercase
- No spaces, use hyphens
- The "title" inside the file can be in the user's language; the file name is always English

## §5. Generation strategy

Phase 3's parallel generation order:

```
Batch 0 (added in V2; only if Phase 2.5 says yes):
  - stage-0-fundamentals/README.md
  - stage-0-fundamentals/prerequisites.md
  - stage-0-fundamentals/recommended-repos.md (1-3 repos from Phase 2C + user evaluation)
  - stage-0-fundamentals/learning-guide.md
  - stage-0-fundamentals/graduation-check.md

Batch 1 (first; others depend on these):
  - meta/learning-path.md
  - meta/teaching-style.md
  - resources/README.md (tiered)
  - reference/glossary.md (build the glossary first)

Batch 2 (depends on Batch 1):
  - stage-1-foundations/*.md (based on Phase 2 web research)
  - reference/code-map.md (based on Phase 2 repo exploration)
  - reference/data-flow.md (based on Phase 2 end-to-end tracing)

Batch 3 (depends on previous two):
  - stage-2-architecture/00-overview.md
  - stage-2-architecture/01-{module}.md ... N-{module}.md

Batch 4 (the heaviest):
  - stage-3-walkthroughs/{N}-{topic}.md
  - After each piece, append to all three exercises/ files

Batch 5 (last; depends on prd.md):
  - stage-4-applied/*.md (only if user provided prd)

Finally:
  - README.md (root, as index)
  - meta/progress.md (empty initial template)
```

Files within a batch can be generated in parallel (independent Write calls).
